# KatelyaTVLocal Codex 项目经验包

生成时间：2026-06-17

适用范围：本文件只适用于 `KatelyaTVLocal`。全局协作底线仍以 `G:\Xiqiuqiu_space\All_Projects\week-review\docs\codex-experience-playbook.md` 为准；本文件记录播放器、搜索、Cloudflare/D1、线路测速和项目交互等项目级经验。若本文件与用户当前明确指令冲突，以用户当前指令为准。

## 一、执行经验总结

### 1. 搜索质量要先追真实下游源

搜索返回大量无关资源时，先检查 `/api/search` 的请求参数、合并逻辑和每个下游接口的实际响应，不要先改排序。此前 `1080zyku` 的 `api_mac10.php` 会忽略 `wd` 返回整页目录，再被项目的额外分页和扁平合并放大，根因在源行为和相关性过滤边界。

调试搜索问题时优先使用真实页面、浏览器网络请求和服务端 API 样本；`curl.exe` 在 Windows 上比部分 PowerShell Web 请求更稳定。

### 2. HLS 广告识别有“清单可见”边界

广告过滤问题要先区分 manifest 可见广告和内容级广告。若广告片段在同一主内容目录、没有 discontinuity、没有广告标签、时长也像正常内容，当前 HLS 清单分析器可能无法可靠识别。

用户给出精确时间窗时，要把时间窗映射回真实媒体分片。例如 17:36-17:52 这类反馈，必须追到对应 m3u8 分片、路径、时长和规则命中情况，再判断是规则缺失、阈值边界还是内容级广告。

### 3. iOS 使用 Apple MMS Runtime 直连，不重新引入代理播放列表

iOS 之前尝试过代理播放列表重写，已暴露连接不稳定和播放状态同步失灵问题。新的修复和方案设计不能把代理播放列表重新混入 iOS 实际播放主路径；iOS/iPadOS 17.1+ 应保持 hls.js + ManagedMediaSource 直连，必要时用旁路分析、日志观测和保守跳过策略。

桌面、Android 与 iOS 现共用 hls.js 恢复模型，但 ManagedMediaSource 仍有 Apple 平台边界。桌面 Chrome 回测可以证明共享逻辑，不能直接证明 iOS 真机行为已经修好。

### 4. 播放卡顿恢复要保护用户动作

卡顿恢复、自动切源、加载超时和拖拽恢复都必须避开用户手动暂停、手动拖动进度、主动切源等操作。不要因为 `waiting`、`stalled` 或计时器残留，就把用户动作误判成播放卡死。

恢复逻辑要清楚区分：起播前 manifest/network fatal、播放中卡顿、拖拽后的正常缓冲、浏览器自动播放限制、当前源真实不可用、切源后恢复时间点。失败升级为切源前，应确认当前恢复策略已经不能解决问题。

### 5. 播放源策略以快速起播和渐进检测为核心

不要追求进入播放页前一次性完整测速。更合理的策略是：读取本地记忆和 D1/后端排序快速起播；起播后后台低并发、低带宽测试候选源；只有当前源真实不可用、严重卡顿或用户主动切换时，才让测速结果影响播放。

当前正在播放的线路应固定在第一位并保持“当前”选中态；如果用户要求“保持现状”“不扩大测速范围”“沿用综合分”，就不要扩大探测预算或重写排序模型。

### 6. 线路面板要解释测速来源

线路面板里的“后端测速”“本机测速”“历史记忆”“待检测”“检测中”“不可切换”不是同一个状态。后端首片速度、浏览器真实测速、用户点击后的单源测速、D1 历史反馈都要在 UI 和逻辑上区分清楚。

“未知测速”不等于源不可用；“检测中”也不能长期悬挂。缺失数据时要显示可理解状态，并允许用户在合理边界内主动尝试。

### 7. D1 调试和配置缓存要追热路径

Cloudflare/D1 问题要先确认调用链。`SELECT config FROM admin_config WHERE id = 1` 高频时，热路径主要可能在服务端搜索、详情、AI 找片、配置读取和反馈链路，不要先用浏览器缓存解释服务端热查询。

更合适的缓存边界通常是：请求内去重、D1 模式 Edge 模块短 TTL 缓存、显式失效入口、返回副本避免跨请求共享对象被污染。缓存优化必须能解释何时更新、谁触发失效、历史请求是否会读到旧对象。

### 8. 播放历史回放要设计 fallback

较早播放历史回放可能找不到对应资源。分析时要追历史记录保存的 source/id/title/year、详情页 fallback、搜索 fallback、最终播放源列表和线路面板是否一致。未找到匹配结果的 UI 应符合当前产品风格，并给用户可继续操作的路径。

### 9. GitHub CI 要对应真实部署风险

PR failing 时，先读具体 failing job 和日志，再解释它覆盖的风险。Docker Build & Test 不只是“跑构建”，它能覆盖 Dockerfile、Corepack/pnpm 版本、lockfile、Next standalone 输出、运行时文件复制等真实部署问题。lint、typecheck、test、Docker build 各自价值不同。

## 二、个人偏好与产品理念

- 用户重视真实播放行为、搜索质量、UI 响应、线路可用性和部署后的回测，不接受只看代码命名或只跑局部命令就下结论。
- 调试播放问题时，要优先使用用户给出的真实 URL、登录信息、预览地址、设备平台、时间点和 D1 调试日志。
- 产品体验上，快速起播、不中断正常播放、失败时可理解、用户可主动切换，比后台打分模型本身更重要。
- 自动过滤、自动跳过、自动切源都必须保守，避免误删内容、误跳剧情、误伤用户暂停和拖拽。
- 管理员调试能力要有开关和历史记录；普通用户不应被调试入口和日志噪声打扰。
- 路线图和优先级必须基于已实现功能和浏览器实测体验，不要只按文档或功能名评估。

## 三、可复用规则清单

1. 播放广告问题先从 `analyzeM3U8AdCandidates`、`filterAdsFromM3U8` 和真实 m3u8 分片入手。
2. 搜索无关结果先检查 `/api/search` payload、下游源响应和相关性过滤，再考虑排序。
3. iOS 播放问题按 Apple MMS Runtime 设计；共享 hls.js 测试不能替代 ManagedMediaSource 真机边界。
4. iOS 不重新引入代理播放列表作为实际播放主路径。
5. 自动切源和卡顿恢复必须排除用户暂停、拖拽、主动切源等人工操作。
6. 首源选择优先快速起播，后台渐进测速；测速结果不打断正常播放。
7. 当前播放源固定优先展示；保持既有综合分和探测预算，除非用户明确要求重设模型。
8. 线路面板必须区分后端测速、本机测速、历史记忆和缺失状态。
9. D1 热查询先追服务端热路径，再决定请求内去重、Edge 缓存或浏览器缓存。
10. Cloudflare 预览和线上问题要区分本地构建、Pages preview、真实部署和当前提交。
11. AI 找片和搜索结果问题要看接口返回与候选过滤，不只看最终 UI。
12. 计划文档保存到 `docs/plans` 时，要做单文件格式检查。
