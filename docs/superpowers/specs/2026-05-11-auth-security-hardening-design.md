# KatelyaTV 认证与权限安全加固设计

## 背景

当前项目已经具备基础的登录、角色区分和管理员能力，但认证体系仍然存在几类高风险问题，会直接影响账号安全和权限边界。已确认的核心问题包括：仓库内存在认证密钥、用户密码以明文方式保存、登录 Cookie 可被前端脚本读取、用户被封禁或改密后旧登录不会立即失效，以及个别管理员接口仍然使用有副作用的 `GET` 请求。这些问题不需要同时被攻击才会出事，任何一个单独成立都足以让现有权限模型变得脆弱。

本设计不追求一次性重写整套用户系统，而是采用两阶段方案。第一阶段优先处理最危险、最容易被直接利用的问题，并允许升级后所有用户重新登录。第二阶段在第一阶段稳定后，继续补强会话失效、CSRF 保护、权限收口和审计能力。这样做的目的是控制改动面，避免把登录、播放、管理后台和存储层同时搅乱，同时确保每一阶段结束后都能独立交付、独立验证和独立回退。

## 目标与范围

本方案的目标是把当前认证体系从“可用但脆弱”提升到“可用且具备基本防护”，并保持现有业务路径尽量少受影响。第一阶段完成后，仓库中不再包含认证密钥，数据库中不再保存明文密码，所有旧会话在升级后失效，新的登录态不再暴露给前端脚本读取。第二阶段完成后，改密、封禁、提权和降权等高风险动作会立即使旧登录失效，管理员敏感接口不再能被简单的跨站请求触发，普通用户改密也不再只依赖“已经登录”这一条件。

本方案不覆盖第三方登录、短信验证码、邮箱验证、多因素认证或复杂风控，也不引入完整的 OAuth/OIDC 体系。项目仍然保留当前的本地部署和 Cloudflare 部署模式，只对现有的用户名密码登录、Cookie 会话、管理员接口和用户管理流程做安全加固。用户已经明确接受升级后需要重新登录，这使得本方案可以清理旧会话格式，而不需要长期维持双格式兼容。

## 方案对比与选择

可选路径有三种。第一种是“只做最小加固”，只修明文密码、Cookie 和仓库密钥问题，优点是快，缺点是旧会话模型和权限判断方式仍然脆弱，后面还得再动一次。第二种是“一次性重构完整认证系统”，优点是最终结构最干净，但改动面过大，会同时触碰登录、中间件、管理接口、用户存储和部署配置，回归成本高。第三种是“分阶段推进”，先止血，再做结构性补强。

本设计选择第三种。理由很简单：当前风险里最危险的部分都集中在“密钥、密码、Cookie、旧会话失效”这几个点上，这些问题可以先在不重做整套业务的前提下修掉。等第一阶段稳定后，再处理会话版本、CSRF、审计和权限统一。这种顺序更符合当前项目状态，也更适合边部署边验证，不会把安全改造变成一次高风险的大手术。

## 第一阶段：高危止血

第一阶段处理五件事。第一，移除仓库中的认证敏感信息。现有 `USERNAME`、`PASSWORD`、cron token 等值不再写进提交文件，改为从 Cloudflare secret 或本地未提交环境文件读取。第二，引入密码哈希存储。第一阶段不强制新增表字段，而是优先复用现有密码存储位，把明文值替换成可识别的哈希串；新注册、改密和管理员新增用户都只写哈希，不再写明文。第三，升级登录 Cookie。新的会话 Cookie 设置为 `httpOnly`、生产环境 `secure`、`sameSite=lax`，不再允许前端脚本读取。第四，替换旧会话格式。升级后统一拒绝旧格式 Cookie，要求所有用户重新登录。第五，为站长凭证和签名密钥分离职责：站长登录凭证单独存储，Cookie 签名使用独立的 `AUTH_SIGNING_SECRET`，不再复用同一个密码字符串承担多个安全角色。

密码哈希算法选择 Web Crypto 兼容方案，而不是依赖 Node 专用库。推荐使用 `PBKDF2-SHA256` 配合每用户独立随机盐和足够高的迭代次数，并把算法元信息编码进哈希串本身，为后续升级留出空间。这样可以兼容 Cloudflare Edge 运行时，减少平台差异。第一阶段不会立即删除旧密码存储位，而是先完成读旧写新和批量升级，再在后续阶段视需要演进出更细的密码字段。

## 第二阶段：结构性补强

第二阶段处理“登录之后还能否被长期滥用”的问题。核心做法是引入会话版本或状态戳。每个用户保存一个 `session_version` 或等价字段，Cookie 会话里也带上同样的值。每次改密、封禁、解封、角色调整、管理员重置凭证时，都让服务端增加用户的会话版本。中间件在放行前不仅验证签名，还会检查当前数据库中的版本是否与 Cookie 一致。这样旧登录会立即失效，不再需要等到 Cookie 自然过期。

同时，第二阶段会把敏感管理接口统一收口。所有会修改状态的管理员接口改为 `POST` 或更明确的方法，不再允许副作用型 `GET`。针对管理员操作增加 CSRF 防护，可以采用双提交 token 或基于服务端生成的表单 token，只要最终做到“浏览器自动带上的 Cookie 本身不足以触发敏感操作”。普通用户改密码时要求提交旧密码，避免“只要拿到当前登录态就能改密码”。另外补一层基础限流和审计记录：登录失败次数、管理员用户变更、封禁与解封、密码修改，都要留下最少可追溯信息，便于后续排查。

## 认证、权限与数据结构设计

新会话模型采用“最小负载的签名 Cookie”，而不是把完整用户状态塞进 Cookie。Cookie 里只保留必要字段，例如用户名、角色、会话版本、签发时间和过期时间；服务端通过 `AUTH_SIGNING_SECRET` 验签，并在数据库模式下查询用户当前状态。这样做的目的不是把系统做成重量级会话中心，而是在维持当前部署简单性的前提下，补上旧系统最大的缺口。对于本地 `localstorage` 模式，需要重新评估是否继续保留。如果保留，也只能保留在明确的单机调试场景，并降低能力，不允许它继续成为正式安全模型的一部分。

数据库层在第二阶段可继续演进，例如补上 `session_version`、`password_changed_at`，必要时再补 `last_login_at` 和 `failed_login_count`。第一阶段为了降低迁移风险，优先复用现有 `password` 存储位，只调整值的格式和校验方式。管理员角色和封禁状态仍可沿用现有用户配置，但权限判断要收敛到统一入口，避免各接口各写一套规则。最终目标是：路由层只关心“这个请求需要什么权限”，而不直接拼接各类用户名或角色特例。

## 迁移、发布与回退

发布顺序必须明确。第一步，在 Cloudflare 和本地环境中准备新的 secrets，但暂不切流。第二步，执行数据库迁移，新增哈希和会话字段。第三步，发布同时支持“新字段写入”和“旧 Cookie 拒绝”的版本，并主动清理旧认证 Cookie，让所有用户重新登录。第四步，验证登录、播放、收藏、继续观看、管理员用户管理和密码修改等主流程。第五步，在确认第一阶段稳定后，再继续上线第二阶段的会话版本校验、CSRF 保护和审计能力。

回退策略也要提前定义。若第一阶段发布后出现登录大面积失败，可以回退到“仍使用新 secrets、但暂时放宽新旧 Cookie 兼容”的临时版本，但不能回退到再次接受明文密码入库的版本。数据库一旦完成哈希迁移，就不应再写回明文。第二阶段若出现兼容性问题，可以单独关闭 CSRF 或会话版本严格校验，但不回退第一阶段已经完成的密钥隔离、哈希密码和 `httpOnly` Cookie。

## 测试与验收

第一阶段的完成标准是：仓库中不再提交认证密钥；数据库中新增用户和改密后的用户不再保存明文密码；旧 Cookie 在升级后失效；新的 Cookie 无法被前端脚本直接读取；登录、退出、普通用户访问、管理员访问和播放主流程都能继续工作。验证方式包括单测、接口测试和浏览器实测，尤其要在实际页面里确认登录后可用、退出后失效、清 Cookie 后需要重新登录。

第二阶段的完成标准是：用户改密码后旧设备登录立即失效；用户被封禁后旧登录立即失效；管理员敏感接口不能通过直接访问链接触发；普通用户改密码必须提供旧密码；登录失败存在限流或至少有明确的失败次数控制；关键敏感操作有可追溯记录。测试上除了已有回归，还要补针对封禁、改密、降权、CSRF 和旧会话复用的反向验证，确保系统在异常路径上也按预期拒绝请求。
